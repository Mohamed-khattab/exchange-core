Executive Summary

This document defines the requirements, architecture, and delivery plan for a production-grade central limit order book (CLOB) matching engine implemented in Go. The system accepts order submissions from market participants, maintains per-instrument price-time priority queues, matches crossing orders into trades, and broadcasts execution reports in real time.

The engine is designed to be embedded in a larger trading platform or operated as a standalone microservice. It prioritises deterministic latency, correctness of fill logic, and horizontal scalability by instrument shard.

Goals & Non-Goals

In Scope

  • ✓
    Price-time priority matching — strict FIFO within each price level; maker's price always wins on crossing orders.
  • ✓
    Four order types — LIMIT, MARKET, IOC (Immediate-or-Cancel), FOK (Fill-or-Kill), with correct partial fill semantics for each.
  • ✓
    Multi-instrument support — each instrument runs in its own goroutine with a dedicated order book, eliminating cross-instrument lock contention.
  • ✓
    REST API — order submission, cancellation, order book depth snapshots, and per-instrument statistics over HTTP/JSON.
  • ✓
    Fixed-point arithmetic — prices and quantities represented as int64 scaled to 8 decimal places, eliminating floating-point rounding errors.
  • ✓
    Volume-weighted average fill price — VWAP tracked per order across partial fills.
  • ✓
    In-memory metrics — per-instrument counters for order volume, trade count, cancellations, and latency histograms.
  • ✓
    Graceful shutdown — SIGINT/SIGTERM handling with a 10-second drain window for in-flight requests.

Out of Scope (v1.0)

  • ✗
    Persistence / crash recovery — the book is in-memory only; orders are lost on restart. WAL or event sourcing is a v2 item.
  • ✗
    WebSocket real-time feed — the WS endpoint is stubbed; full market data dissemination is v2.
  • ✗
    Authentication & authorisation — no client identity enforcement in v1; intended for trusted internal networks.
  • ✗
    Risk checks — pre-trade risk controls (position limits, credit checks) are delegated to the upstream gateway.
  • ✗
    FIX protocol — binary/FIX connectivity is a v2 integration point.

Architecture

The engine uses a shared-nothing, per-instrument actor model. Each instrument (e.g. BTC-USD) owns a single goroutine that sequentially processes commands from a buffered channel. This design eliminates mutex contention between instruments and gives each book deterministic, single-threaded execution semantics.

Order Book Design

Each side of the book (bids, asks) is maintained as a sorted slice of price levels, each level holding a doubly-linked FIFO list of orders. Binary search locates price levels in O(log N) time; insertion maintains the sorted invariant.

Data Structures

StructurePurposeComplexity
priceTree (sorted slice) Ordered collection of price levels per side O(log N) lookup, O(N) insert/delete for N price levels
PriceLevel Aggregated quantity + FIFO order queue at a single price O(1) enqueue, dequeue, peek
orderMap (map[uint64]) Fast O(1) order lookup by ID for cancellation O(1) get, insert, delete

⚡ Production note: for books with >5,000 distinct price levels (e.g. tick-by-tick equity books), replace the sorted slice with a red-black tree (e.g. emirpasic/gods). For crypto derivatives and FX — typically <500 active levels — the slice is optimal.

Matching Logic

Priority Rules

The engine enforces strict price-time priority:

  1. Price priority — the best priced order always matches first (highest bid / lowest ask).
  2. Time priority — among orders at the same price, the earliest submitted order fills first (FIFO).
  3. Maker price wins — on a crossing, the resting order's price is used as the execution price.

Order Type Semantics

TypeBehaviourRemainder handling
LIMIT Matches against any contra-side order ≤ (buy) or ≥ (sell) the order's price Rests in book at specified price until filled or cancelled
MARKET Matches at the best available price(s) on the contra side, walking the book Rejected with status REJECTED if no liquidity exists
IOC Same as LIMIT — matches as aggressively as possible at the specified price Any unfilled remainder is immediately cancelled (never rests)
FOK Scans available liquidity first; only executes if the full quantity can be filled If full fill is impossible: zero trades, order cancelled atomically

Match Loop (Pseudocode)

func matchLimit(order *Order) []MatchResult { for order.RemainingQty() > 0 { contra := book.BestContraSide() if contra == nil || !pricesCross(order, contra) { break // no more matching levels } result := fillAgainst(order, contra.Peek()) results = append(results, result) } if order.RemainingQty() > 0 { book.Add(order) // rest remainder in book } return results }

Functional Requirements

IDRequirementPriority
FR-01 The engine MUST assign a monotonically increasing, globally unique order ID to each submitted order. CRITICAL
FR-02 LIMIT orders MUST match at the resting order's price (maker price priority). CRITICAL
FR-03 Within the same price level, orders MUST match in FIFO (time priority) order. CRITICAL
FR-04 MARKET orders MUST be rejected (status=REJECTED) if no contra-side liquidity exists. CRITICAL
FR-05 FOK orders MUST NOT execute any partial fills — the order is either fully filled or fully cancelled. CRITICAL
FR-06 IOC orders MUST cancel any unfilled remainder immediately after matching. HIGH
FR-07 Each trade MUST record: trade ID, instrument, buy order ID, sell order ID, price, quantity, timestamp, and aggressor side. HIGH
FR-08 The engine MUST track volume-weighted average fill price (VWAP) for each partially or fully filled order. HIGH
FR-09 Order cancellation MUST be O(1) — accomplished via the per-order map, not by scanning the book. HIGH
FR-10 The engine MUST apply back-pressure when an instrument's command channel is full, returning an error to the caller. MEDIUM
FR-11 Order book snapshots MUST be served concurrently with order processing without blocking the matching hot path. MEDIUM
FR-12 The engine MUST support at minimum 4 active instruments simultaneously at launch (BTC-USD, ETH-USD, SOL-USD, BNB-USD). MEDIUM

Non-Functional Requirements

Reliability

  • ◈
    Graceful shutdown must drain in-flight commands within 10 seconds before stopping workers.
  • ◈
    No goroutine leaks — all workers must terminate cleanly on SIGINT/SIGTERM.
  • ◈
    No silent data corruption — price arithmetic must use fixed-point int64 exclusively.

Observability

  • ◈
    Per-instrument latency histograms (10 buckets from <1µs to ≥1s) exposed via /v1/stats.
  • ◈
    Structured log lines for every trade execution and every HTTP request.
  • ◈
    v2: Prometheus metrics endpoint at /metrics for Grafana dashboards.

Deployability

  • ◈
    Single statically-linked binary with zero external runtime dependencies (CGO_ENABLED=0).
  • ◈
    Multi-stage Docker build producing a <20MB scratch image.
  • ◈
    Config via environment variables (future); instruments configured at startup.

Testability

  • ◈
    Unit tests covering all order type semantics: LIMIT cross, LIMIT rest, MARKET no-liquidity, FOK full fill, FOK cancel, IOC partial, order cancellation.
  • ◈
    Go benchmarks for hot paths (match loop, limit order add) included in the orderbook package.
  • ◈
    All tests must pass with go test -race ./... (data race detector enabled).

Risks & Mitigations

RiskImpactLikelihoodMitigation
Memory exhaustion from unbounded open orders CRITICAL MEDIUM Enforce max open orders per instrument and per client via config; add order eviction for expired GTD orders
Channel queue full causes order loss CRITICAL LOW Back-pressure returns error to caller; increase channel buffer (default 10,000); monitor queue depth metric
Clock skew across instances causes incorrect trade timestamps HIGH MEDIUM Use monotonic clock for sequencing; NTP sync enforced via infra policy; logical sequence numbers always primary sort key
Panic in matching goroutine crashes the book CRITICAL LOW Add recover() wrapper in worker run loop; log panic trace; signal health endpoint as degraded; restart worker with current book state
Price level sorted slice degrades at >5k levels HIGH LOW Monitor distinct price levels metric; swap to red-black tree if approaching 5k limit; add alert threshold in v1.1
API misuse causes phantom orders (client sends same order twice) MEDIUM HIGH client_id idempotency key de-duplication (v1.1); currently the engine will accept duplicates as separate orders
